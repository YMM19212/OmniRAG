# 文本去重测试报告

## 1. 这份报告在说什么

这份报告总结了两次文本去重测试的结果：

1. 第一次测试：修复前，系统几乎完全识别不出重复文本。
2. 第二次测试：修复后，系统已经可以稳定识别重复文本，并找到了一个较合适的阈值区间。

这两次测试都基于同一个数据集：

- 数据文件：`dataset/train-0002-of-0724-f139ff49-3de5-4143-8132-bd92c480d313.parquet`
- 测试线：`text`
- 向量模型：`jina-embeddings-v4`

## 2. 第一次测试结果：修复前

结果文件：

- `data/dedup_eval_text_only/summary.json`
- `data/dedup_eval_text_only/threshold_metrics.csv`

测试配置：

- 原始样本数：`2`
- 评测样本数：`10`
- 阈值：`0.82`

结果：

| 指标 | 数值 |
| --- | ---: |
| TP | 0 |
| FP | 0 |
| FN | 6 |
| TN | 4 |
| Precision | 0.0 |
| Recall | 0.0 |
| F1 | 0.0 |

一句话解释：

> 当时系统一个重复样本都没有识别出来，表现为“完全漏判”。

根因：

- `Milvus COSINE` 的返回分数被错误地写成了 `1 - hit.distance`
- 这会把高相似度结果反向压低
- 直接导致语义去重判断几乎全部失效

## 3. 第二次测试结果：修复后

结果文件：

- `data/dedup_eval_text_sweep_fix_v2/summary.json`
- `data/dedup_eval_text_sweep_fix_v2/threshold_metrics.csv`
- `data/dedup_eval_text_sweep_fix_v2/variant_metrics.csv`

测试配置：

- 原始样本数：`6`
- 评测样本数：`24`
- 阈值扫描范围：`0.60 ~ 0.90`
- 步长：`0.05`
- LLM 改写：本轮关闭

最佳结果：

| 指标 | 数值 |
| --- | ---: |
| 最优阈值 | 0.80 |
| TP | 18 |
| FP | 0 |
| FN | 0 |
| TN | 6 |
| Precision | 1.0 |
| Recall | 1.0 |
| F1 | 1.0 |

一句话解释：

> 修复后，当前这组文本样本在 `0.80` 及以上阈值已经可以做到“重复不漏、非重复不误判”。

## 4. 不同阈值下的表现

| 阈值 | Precision | Recall | F1 | 说明 |
| --- | ---: | ---: | ---: | --- |
| 0.60 | 0.7826 | 1.0 | 0.8780 | 召回很好，但误判偏多 |
| 0.65 | 0.8571 | 1.0 | 0.9231 | 比 0.60 更稳 |
| 0.70 | 0.8571 | 1.0 | 0.9231 | 与 0.65 基本一致 |
| 0.75 | 0.9474 | 1.0 | 0.9730 | 已经很接近可用 |
| 0.80 | 1.0 | 1.0 | 1.0 | 当前测试集最佳点 |
| 0.85 | 1.0 | 1.0 | 1.0 | 与 0.80 一样稳定 |
| 0.90 | 1.0 | 1.0 | 1.0 | 与 0.80 一样稳定 |

可以这样理解：

- 阈值越低，系统越容易把“相近文本”当成重复
- 阈值越高，系统越保守
- 在当前这批样本上，`0.80 ~ 0.90` 是最稳的区间

## 5. 这次结果说明了什么

### 5.1 修复是有效的

修复前：

- `Recall = 0.0`
- 系统完全抓不到重复文本

修复后：

- `Recall = 1.0`
- 重复文本已经可以全部识别出来

这说明核心问题不在 embedding 模型，而在去重分数解释逻辑。

### 5.2 当前推荐阈值

如果只看这轮文本测试，我建议：

- 先把文本语义去重阈值设为 `0.80`

原因：

- 比 `0.75` 更稳
- 没有出现误判
- 也没有漏判

### 5.3 结果还不能过度外推

这轮结果很好，但还不能直接说明“真实业务里一定也是 100%”。

原因有三个：

1. 当前样本量还不大，只是第一轮验证。
2. 本轮关闭了 LLM 改写，难样本强度还不够高。
3. 目前只测了文本线，还没有把图文线完整跑通并做同样扫描。

## 6. 这次评测里顺手修掉的一个基准问题

在评测脚本里，早期的 `hard_negative` 有一个标注偏差：

- 当 LLM 关闭时，脚本会直接借用另一个样本的原文
- 这种样本在全局语料里其实是真重复
- 但旧逻辑把它记成了 `False Positive`

这个问题已经修正。现在这类样本会单独标成 `fallback_neighbor`，不再污染指标。

## 7. 当前结论

目前可以下的结论很直接：

- 文本去重链路原本确实有 bug
- 这个 bug 已经修复
- 修复后文本去重能力明显恢复
- 在当前 no-LLM 样本上，`0.80` 是一个合理的默认阈值

## 8. 下一步建议

建议按这个顺序继续：

1. 开启 LLM 改写，再做一轮更难的文本去重评测。
2. 修完图文线后，再做图文联合去重评测。
3. 最后再决定是否把默认阈值正式改到 `0.80`。
